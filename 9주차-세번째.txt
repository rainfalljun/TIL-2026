============================================================
         DynamoDB 강의 요약 정리
============================================================

------------------------------------------------------------
1. RDBMS (관계형 데이터베이스)
------------------------------------------------------------
- 행과 열로 구성된 2차원 테이블 기반의 정형화된 스키마로 데이터 저장
- ACID 트랜잭션 처리
- SQL, Join 등 복잡한 쿼리 처리 가능
- 주요 AWS 서비스: Amazon RDS, Aurora, Redshift
- 주요 사용 사례: 대부분의 애플리케이션 메인 DB

[ACID란?]
- Atomicity  (원자성) : All or Nothing, 전부 성공하거나 전부 실패
- Consistency (일관성) : 트랜잭션 전후 데이터가 논리적으로 올바른 상태 유지
- Isolation   (독립성) : 동시 트랜잭션 간 서로 간섭하지 않음
- Durability  (지속성) : 트랜잭션 결과는 디스크에 영구 저장

------------------------------------------------------------
2. NoSQL (Not Only SQL)
------------------------------------------------------------
- 정해진 스키마 없이 유연하게 다양한 형식의 데이터 저장
- BASE 기반 트랜잭션 처리
- 비교적 간단한 쿼리만 수행 가능
- 확장성(분산 처리)이 뛰어남
- 주요 AWS 서비스: DynamoDB, MemoryDB (Redis, Memcached, Valkey)
- 주요 사용 사례: 캐시, 간단한 데이터 저장, 추천 로직 등

[NoSQL의 종류]
- Key-Value Stores  : Key-Value 쌍으로 데이터 저장
- Document Database : JSON 형식의 문서 형태로 데이터 저장
- Graph Database    : 노드(사람)와 엣지(관계)를 중심으로 데이터 저장 (SNS에 주로 사용)

[BASE란?]
- Basically Available  : 가용성 중심, 다수 스토리지에 분산 저장
- Soft-State           : DB 상태가 느슨하고 유연하게 유지됨
- Eventual Consistency : 즉각적인 일관성이 아닌, 시간이 지나 일관성 수렴

------------------------------------------------------------
3. RDBMS vs NoSQL 비교
------------------------------------------------------------
항목           RDBMS                       NoSQL
--------------------------------------------------------------
데이터 형식    정해진 스키마로 관리         자유로운 데이터 형식
데이터 처리    ACID                         BASE
확장성         확장 어려움                  분산 처리가 비교적 쉬움
쿼리 복잡도    복잡한 쿼리 수행 가능        비교적 간단한 쿼리만 가능
AWS 서비스     RDS, Aurora, Redshift        DynamoDB, MemoryDB 등

4. DynamoDB 핵심 개념
------------------------------------------------------------

[파티션 키 (Partition Key)]
- 모든 테이블에 반드시 하나 존재, 각 항목(Item)의 고유 ID 역할
- 내부적으로 해시(Hash) 처리 → 데이터를 어느 물리 파티션(서버)에 저장할지 결정
- 같은 파티션 키 값끼리는 같은 물리 저장소 파티션에 묶임
- 분산 저장, 성능 균형, 확장성의 장점

예) minji01 → 해시 → 983248123912 → Partition A

[정렬 키 (Sort Key)]
- 파티션 키만 사용하면 한 명당 하나의 항목만 저장 가능
- 동일 파티션 키를 가진 항목을 여러 개 저장하고 정렬하려면 정렬 키 필요
- 기존 테이블에 정렬 키를 추가하는 것은 불가 → 새 테이블 생성 필요

[날짜 타입]
- DynamoDB에는 날짜 타입이 없음
- ISO 8601 형식(예: 2025-05-11T21:00:00)의 문자열로 저장하면 시간순 정렬 가능

5. 실습 순서 요약
------------------------------------------------------------
실습 0  DynamoDB 테이블 생성
        - 테이블명: sgu-202500-user-likes
        - 파티션 키: user_id

실습 1  항목 추가
        - user_id: jihoon07 / product_name: 스탠리

실습 2  항목 편집
        - 항목 선택 → 작업 → 항목 편집

실습 3  정렬 키 추가한 새 테이블 생성
        - 테이블명: sgu-202500-user-likes-time
        - 파티션 키: user_id (문자열)
        - 정렬 키: timestamp (문자열)

실습 4  새 테이블에 항목 추가 (4개 사용자 데이터 JSON 입력)
        - haeri05, minji01, jihoon07, yuna23

실습 5  jihoon07에 항목 추가
        - timestamp: 2025-05-12T18:15:00
        - product_name: 텀블러 악세사리

실습 6  jihoon07 항목 시간순 정렬 조회
        - 쿼리 선택 → 파티션 키: jihoon07 → 정렬 키 기준 오름/내림차순 설정
