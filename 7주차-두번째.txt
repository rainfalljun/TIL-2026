Git 인터페이스 (사용 방법)

CLI (Command-Line Interface): 터미널, 콘솔, 명령 프롬프트(cmd) 등을 통해 명령어를 직접 입력하는 방식. (예: git add, git commit)

GUI (Graphical User Interface): 버튼이나 메뉴 등 그래픽을 통해 마우스로 조작하는 방식. (예: GitHub Desktop, SourceTree)

Git의 기본 개념

DVCS(Distributed Version Control System): 분산 버전 관리 시스템.

소스 코드의 변경 사항을 추적하고 협업 및 복구(어제로 되돌리기 등)를 위해 사용.

초기 계정 설정 (필수 암기)

이름 설정: git config --global user.name "이름"
            (전역)
이메일 설정: git config --global user.email "이메일"

설정 확인: git config --global --list (목록)

설정 삭제: git config --global --unset user.name (또는 user.email)

줄바꿈 호환성(Windows): git config --global core.autocrlf true

.gitignore 작성 규칙 (교수님 강조)

역할: 보안상 중요한 파일이나 불필요한 파일을 Git 관리 대상에서 제외.

: 주석 (설명)
*.log : 특정 확장자 전체 제외

/config.xml : (최상위) 루트 폴더의 해당 파일만 제외

config.xml/ : 해당 루트 포함 그 내부의 모든 내용을 제외

!important.log : 제외 대상 중 특정 파일만 다시 포함 (예외의 예외)

Git의 3단계 흐름

Working Directory: 실제 작업 공간 (내 컴퓨터 폴더)

Staging Area: 저장할 파일들을 임시로 담는 바구니 (git add 명령어 사용)

Repository: 최종적으로 변경 사항이 영구 기록되는 곳 (git commit 명령어 사용)

파일의 상태 (Unmodified 중심)

Untracked: 한 번도 add 되지 않은 상태.

Unmodified: commit 직후의 상태. 저장소의 내용과 내 컴퓨터의 내용이 완벽히 일치하는 깨끗한 상태.

Modified: 마지막 저장 이후 파일이 수정된 상태.

Staged: git add를 통해 Staging Area에 올라간 상태.



주요 명령어

git status: (현재) 파일들의 상태 확인 (수정됨, 스테이징됨 등).

git log: (과거) 지금까지 만든 커밋 목록 확인.

git add .: 현재 폴더의 모든 변경사항을 스테이징 영역으로 이동.

git commit -m "메시지": 스테이징 영역의 파일을 저장소에 기록.

git reset HEAD^: 가장 최근에 한 커밋(1개)을 취소.

git reset HEAD 파일명: 스테이징된 파일을 다시 내리기(Unstage).

git restore <파일명>: 작업 디렉토리에서 수정된 내용을 취소하고 마지막 커밋 상태로 되돌립니다.

git restore --staged <파일명>: 스테이징 영역에 올라간 파일을 다시 워킹 디렉토리로 내립니다(Unstage).

Git 저장소를 처음부터 새로 만들거나, 현재 폴더를 Git 관리 폴더로 만들고 싶을 때 가장 먼저 실행해야 하는 명령어는 무엇인가요?

git init

[Git 중급/고급 & GitHub 핵심 요약본]

1. Git과 GitHub 개념
- Git: 내 컴퓨터에서 소스코드 버전(역사)을 관리하는 도구 (분산 버전 관리 시스템)
- GitHub: Git으로 관리한 프로젝트를 올리는 온라인 저장소 (협업 및 호스팅 플랫폼)

2. 브랜치(Branch) 필수 명령어 (교수님 강조)
- 목록 확인: git branch
- 생성: git branch 브랜치명
- 이동: git switch 브랜치명
- 생성과 동시에 이동: git switch -c 브랜치명
- 이름 변경: git branch -m 기존이름 새이름
- 삭제: git branch -d 브랜치명

3. 브랜치 병합 (중요)
- Merge: 두 브랜치의 기록을 합치는 것. main으로 이동 후 'git merge 대상브랜치'
- Rebase: 브랜치의 시작점(공통 조상)을 옮겨서 선형적인 이력을 만드는 것
- 충돌 발생 시: 파일 수정 -> git add 파일명 -> git rebase --continue 순서로 진행

4. 원격 저장소 연결 및 사용 (Push/Pull/Clone)
- 원격 저장소 복제: git clone 저장소주소
- 원격 저장소 연결: git remote add origin 저장소주소
- 로컬 내역 전송: git push origin 브랜치명
- 원격 내역 가져오기: git pull origin 브랜치명

5. GitHub 액세스 토큰 (Access Token)
- 보안 강화로 비밀번호 대신 사용함
- 경로: Settings -> Developer settings -> Personal access tokens (classic)
- 주의: 발급 시 토큰값을 복사해서 따로 저장해야 하며, push 시 비밀번호 칸에 입력

6. 되돌리기 (Reset vs Revert)
- Reset: 특정 커밋 시점으로 완전히 돌아감 (이후 기록 삭제됨)
- Revert: 이전 상태로 되돌리는 '새로운 커밋'을 생성함 (기존 기록 유지됨)

7. 암기용 핵심 루틴
- 코드 수정 후: git add . -> git commit -m "메시지" -> git push origin 브랜치명
- 협업 전 필수: 항상 git pull을 먼저 해서 원격의 최신 상태를 받아올 것

[AWS 아키텍처 시험 직전 요약본]

1. AWS 글로벌 인프라 (필수 암기)
- 리전(Region): 전 세계에 흩어져 있는 물리적 서버 위치. 서울 리전 코드는 ap-northeast-2.
- 가용영역(AZ): 리전 안에 있는 데이터 센터 묶음. 하나의 리전은 최소 2개 이상의 가용영역으로 구성됨.
- 가용영역 위치: 각 계정마다 AZ의 이름과 실제 물리적 위치는 다르게 매핑됨(보안 및 부하 분산 목적).
- 엣지 로케이션: 사용자에게 콘텐츠를 빠르게 전달하기 위한 캐싱 거점(CloudFront 서비스에 사용).

2. 글로벌 서비스 vs 리전 서비스 (구분 필수)
- 글로벌 서비스: IAM, Route 53, CloudFront, WAF, AWS Billing (전 세계 공통 적용).
- 리전 서비스: EC2, S3, RDS, VPC (특정 리전 내에서 생성 및 사용).

3. ARN (Amazon Resource Name)
- AWS의 모든 리소스에 부여되는 고유 식별자.
- 형식 예시: arn:partition:service:region:account-id:resource-id

4. VPC (Virtual Private Cloud) 및 구성 요소
- 정의: AWS 내 나만의 가상 네트워크 공간.
- 구성 요소(포함되는 것): 서브넷, 인터넷 게이트웨이, 라우팅 테이블, 보안 그룹, 네트워크 ACL, NAT 게이트웨이, VPC 엔드포인트.
- 주의: AWS 서비스 구조도 첫 번째 줄에 나오는 핵심 개념이므로 VPC가 가장 기본 네트워크 단위임을 숙지.

5. IAM (Identity and Access Management)
- 핵심: 누가(인증) 무엇을 할 수 있는지(권한) 관리.
- 사용자: 실제 사람 또는 앱.
- 그룹: 사용자들의 집합. 권한 부여 편의성.
- 역할: 서비스나 사용자에게 임시로 권한을 부여할 때 사용(가장 보안상 안전).
- 정책: JSON 형식으로 작성된 권한 정의 문서.
- 보안 원칙: 루트 계정 사용 자제, MFA 설정 필수, 최소 권한 원칙 준수.

6. 기타 핵심 체크
- 보안 그룹: 인스턴스(서버) 단위의 보안(VPC 내부 시큐리티 핵심).
- AWS 서비스 구조: 인프라 - 기초 서비스(컴퓨팅, 스토리지, 네트워크) - 응용 서비스 순서로 구성됨.

[시험장 직전 핵심 요약본]

1. 클라우드 컴퓨팅
- 이점: 자본 비용을 가변 비용으로 대체, 규모의 경제로 인한 이득, 결과 용량 추정 필요 없음, 속도 및 민첩성 개선, 데이터 센터 운영 및 유지 관리 비용 절감, 몇 분 만에 전 세계 배포
- 온프레미스: 기업이 자체적으로 서버를 보유하고 물리적으로 직접 운영하는 방식
- 서비스 유형: IaaS(인프라 중심), PaaS(개발 플랫폼 중심), SaaS(완성된 소프트웨어 중심)

2. S3 기초 및 특징
- 객체 스토리지: 파일 저장 전용이며 애플리케이션 설치 불가
- 내구성: 99.999999999% (11 9's) 제공
- 가용성: 필요할 때 언제든 데이터에 접근할 수 있는 성질
- 데이터 복제: 최소 3개의 물리적으로 분리된 가용 영역(AZ)에 복제됨
- 버킷 특징: 전 세계 고유 이름을 가져야 하며 생성 후 이름 변경 불가
- 버킷 정책: JSON 형식으로 권한 정의
  * Allow: 특정 작업 허용 (예: GetObject는 읽기, PutObject는 쓰기)
  * Deny: 특정 작업 거부 (거부가 항상 허용보다 우선함)

3. EC2 및 보안
- 보안 그룹(Security Group): 인스턴스 앞단의 가상 방화벽. 인바운드와 아웃바운드 규칙 설정. 허용 규칙만 작성 가능
- 키페어: 인스턴스 접속을 위한 인증 수단 (.pem 또는 .ppk)
- CloudWatch: CPU 사용량 등 지표를 모니터링하고 경보를 설정하는 서비스
- 인스턴스 상태: running(실행), stopped(중지), terminated(종료), pending(대기) 등

4. AWS CLI 및 자동화
- UserData: EC2 부팅 시 자동으로 실행되는 초기 구성 스크립트
- 주요 명령어:
  * aws s3 mb s3://버킷명 : 버킷 생성
  * aws s3 ls : 버킷/객체 목록 확인
  * aws s3 cp 로컬파일 s3://버킷명 : 업로드
  * aws s3 cp s3://버킷명 로컬경로 : 다운로드
  * --recursive : 폴더 단위 작업 시 사용하는 옵션 (마지막 4페이지 핵심)

5. 암기 필수 항목
- S3 내구성 수치: 11개의 9 (99.999999999%)
- S3 버킷 정책 필수 요소: Version, Statement, Effect, Principal, Action, Resource
- 보안 그룹 특징: 생성 후 이름 편집 불가, 기본적으로 모든 아웃바운드 허용 및 모든 인바운드 차단 상태에서 시작함
